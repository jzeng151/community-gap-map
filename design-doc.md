## Problem Statement

NYC residents in need have no single place to see what community services exist near them, whether those services are currently open and available, and what gaps exist where no service covers a need. Three partial solutions exist in isolation:

- **Opportunity Finder** — a search tool for resources
- **Community Pulse** — an anonymous board where residents post unmet needs
- **Community Offerings** — a bulletin board of available services

Together, they form a living map of community need vs. supply for NYC: residents can find help, and organizers can see where help doesn't exist.

## What Makes This Cool

The synthesis is the product. A neighborhood with 8 food needs and 1 food pantry tells a story that neither dataset tells alone. No existing tool combines demand signal (what people need, anonymously posted) with supply (what's open, by whom, and where) on a single navigable map.

The closest analogs are 211 (static, top-down, no community voice) and Nextdoor (community voice, no structure, no gap analysis). This sits between them — structured enough to be useful, community-powered enough to feel alive.

**EUREKA:** Every existing community resource tool is built from the supply side because funders and city agencies are the buyers. The community isn't the buyer — they're the data subject. This is the first tool where NYC residents are both the primary user *and* the primary data contributor.

## Constraints

- NYC-focused (not national, not generic)
- Primary user: residents actively seeking help right now
- Secondary user: community organizers and mutual aid coordinators seeing the gap pattern
- Pulse posts must be fully anonymous — stigma is a real barrier to posting
- Availability status is the #1 question — a resident in crisis who arrives at a closed location has been actively misled, so static data must always show a freshness indicator

## Premises

1. **Needs posts must be anonymous** — stigma is real; identity-free posting is a core design requirement
2. **Seed from NYC Open Data only** — 211 data requires a data-sharing agreement with United Way of NYC and is not freely available; do not depend on it for launch. Bootstrap from NYC Open Data Facilities/HRA datasets.
3. **Finder is primary; gap view deferred to v2** — gap visualization requires meaningful Pulse signal density and will be built once that data exists
4. **"Is it open right now?" is the #1 UX question** — availability status must always display a data freshness timestamp; never show open/closed without "data as of [date]"

## Approaches Considered

### Approach A: Seeded Map + Anonymous Pulse (CHOSEN)
Bootstrap from NYC Open Data → display on interactive map with color-coded pins → add anonymous need submission form. Finder works immediately. Gap view added in v2.

**Effort:** S-M — approximately 2-4 weeks solo
**Risk:** Low

**Pros:** Finder works immediately with real data. Proves the Pulse concept. Zero org onboarding required.
**Cons:** Availability status is static (set at import time). Gap view deferred.

### Approach B: Full Dynamic System
Multi-source pipeline + real-time availability + gap heatmap + org self-service dashboard.
**Effort:** XL · **Risk:** High — requires org adoption before availability data is meaningful

### Approach C: SMS-First Gap Finder
Text "NEED food 11221" → 3 nearest open resources. Web map for organizers.
**Effort:** M · **Risk:** Med — SMS cost at scale; harder to show the map "whoa"

## Recommended Approach

**Approach A — Seeded Map + Anonymous Pulse.**

The gap overlay is deferred to v2. Reason: it requires Pulse signal density to be useful (target threshold for v2: 50+ signals across ≥ 3 boroughs). Building it in v1 before that data exists produces an empty visualization that undermines trust.

---

### Map Design (v1)

**Pin color scheme by provider type (use distinct hues, not gray shades):**
- Blue = Government
- Green = Nonprofit
- Orange = Mutual Aid
- Any pin type + dashed outline = closed or outside posted hours

Rationale: monochromatic gray shades are inaccessible (low-contrast screens, visual impairments) and undermine the stated UX priority of color as the primary information hierarchy.

**Availability status:**
- In v1, `availability_status` is a **static enum** (open / closed / unknown) set at import time based on posted hours at time of seed. It is NOT computed dynamically at request time.
- The "Available now" toggle filters on this static field. The toggle label must read **"Open per listed hours"** — not "Open now" — to set accurate expectations.
- Every pin tooltip displays: `"Hours as of [imported_at date] · Flag if inaccurate"`. This is mandatory — never display availability without it.

**Anonymous need-dots (Pulse):**
- Rendered as small semi-transparent circles, visually distinct from service pins
- Position: neighborhood centroid + random jitter of ±200m; for neighborhoods < 0.5 km², jitter radius is increased to the neighborhood's bounding box diagonal (the more conservative formula — this is the authoritative spec)
- At map zoom < 12: cluster into count bubbles. At zoom ≥ 12: show individual dots.
- Hovering a need-dot shows: category + neighborhood name only — **never free text**

**Gap overlay (v2 only):** Do not ship in v1. Deferred until Pulse data reaches threshold.

**Legend:** always visible, upper-left. Shows pin colors by provider type + availability symbols.

**FAB:** "Report an unmet need" — bottom-center of map.

---

### Mobile Layout

The desktop layout (map 60% width + fixed 300px sidebar) does not work on mobile. Mobile spec:

- **Map:** full-screen, with search bar overlaid at top as a floating pill
- **Filter bar:** horizontal scroll below search pill; chips scroll off-screen
- **Sidebar:** replaced by a **bottom sheet** (collapsed to a handle at 15% screen height; drag up to 60% to see results; full-screen for the Pulse form)
- **Results:** within the bottom sheet, same sort order as desktop
- **FAB:** stays bottom-center, above the collapsed sheet handle

Target: mobile-first implementation. Desktop sidebar is an enhancement for larger screens.

---

### Finder UX

**Search:** free-text input (service type, neighborhood name, or zip). Geo-search via PostGIS `ST_DWithin` against the `offerings` table.

**Filters:**
- Category chips: Food, Housing, Healthcare, Childcare, Legal, Jobs
- Provider type chips: Government, Nonprofit, Mutual Aid
- "Open per listed hours" toggle

**Filter logic:**
- Default (no chips selected): show all results — no filter applied
- Category AND provider type: e.g., "Food" + "Nonprofit" → food nonprofits only
- Multiple categories selected: OR between them
- Multiple provider types selected: OR between them
- Combined SQL: `WHERE (category IN [...]) AND (provider_type IN [...])`

**Results sort:** within each group (open first, then closed), sort by `ST_Distance(lat_lng, user_location) ASC`. No relevance ranking in v1.

### Pulse — Anonymous Need Submission

**Form fields:**
1. Category: dropdown (Food, Housing, Healthcare, Childcare, Legal, Jobs)
2. Description: free-text textarea, max 280 chars
3. Location: **dropdown of NYC neighborhood names** (~200 neighborhoods, curated list) — not free-text, because a free-text field allows exact address submission which breaks the anonymity guarantee
4. Submit: "Submit anonymously"

**What description text is shown and where:**
- On the map (hover): category + neighborhood name only — description is never shown on the map
- In the sidebar Pulse feed: category, neighborhood, time ago — description is never shown in the feed either
- Description text is stored server-side for moderation review only; it is never displayed to any end user

Rationale: free-text descriptions can contain identifying information. Displaying them publicly — even without a name — is a privacy risk for the target population.

**Anonymity guarantees:**
- No login, name, or email collected
- Location stored as neighborhood centroid from dropdown → server-side centroid lookup; the user's typed text is never geo-processed
- The neighborhood label displayed in the Pulse feed is the verbatim dropdown selection (e.g., "Bed-Stuy"), not a reverse-geocode of the jittered coordinate
- **Neighborhood centroid dataset:** use NYC Planning's Neighborhood Tabulation Areas (NTAs) from NYC Open Data (`geo_export` layer). NTAs provide official boundaries and centroids for ~200 named neighborhoods. The dropdown list is derived from this dataset; the server-side centroid lookup maps the selected NTA name to its centroid lat/lng.
- IP address used only for rate limiting; not persisted after request processing
- For neighborhoods < 0.5 km², jitter radius is the neighborhood's bounding box diagonal (same formula as map design — bounding box diagonal is authoritative; the ±200m and the bounding-box-diagonal are not two separate formulas, the latter supersedes the former for small neighborhoods)

**Spam and abuse mitigation:**
- IP rate limiting: max 3 submissions per IP per hour
- Implementation: **Vercel middleware + Upstash Redis** shared counter (required — serverless function instances do not share in-memory state; a shared Redis store is necessary for rate limiting to work correctly across concurrent deployments)
- Honeypot hidden field for bot detection
- Flag button visible on each Pulse post in the sidebar
- Flag increments use an atomic `UPDATE pulse SET flag_count = flag_count + 1 WHERE id = $1 RETURNING flag_count` — the Postgres trigger fires only when the returned value is exactly 3, preventing duplicate webhook calls from concurrent requests
- Posts with `flag_count >= 3`: a Postgres trigger sets `status = 'hidden'` and fires a Supabase webhook → email notification to the product owner. Hidden posts are excluded from the public feed query at the database level (`WHERE status = 'visible'`).
- **Moderation queue owner:** the product owner reviews hidden posts via a Supabase Table Editor saved view filtered to `status = 'hidden'`. Target review window: 24 hours. No AI moderation in v1.

### Data Model (v1)

**offerings table:**
```sql
id            uuid primary key,
name          text not null,
category      text not null,  -- enum: food/housing/healthcare/childcare/legal/jobs
provider_type text not null,  -- enum: gov/npo/mutual-aid
address       text,
location      geography(Point, 4326),  -- PostGIS point
hours_json    jsonb,          -- raw hours from Open Data (stored but not evaluated in v1)
availability_status text not null default 'unknown',  -- enum: open/closed/unknown (set at import)
data_source   text,
imported_at   timestamptz not null,
flagged       boolean default false
```

**pulse table:**
```sql
id            uuid primary key,
category      text not null,  -- enum: food/housing/healthcare/childcare/legal/jobs
description   text,           -- max 280 chars; never displayed publicly; moderation only
neighborhood  text not null,  -- verbatim dropdown selection, displayed in feed
location      geography(Point, 4326),  -- neighborhood centroid + jitter; never shown raw
created_at    timestamptz not null,
flag_count    int default 0,
status        text default 'visible'  -- enum: visible/hidden (set to hidden when flag_count >= 3)
```

No PII stored in either table. `pulse.location` is always a jittered neighborhood centroid, never user-derived.

### Data Pipeline

**Source:** NYC Open Data Facilities & Providers API + HRA Social Services dataset

**One-time seed (v1):** Import script runs once to populate `offerings` table at launch. Sets `availability_status` from hours field at time of import. `imported_at` set to import timestamp (displayed to users as data freshness date).

**Deduplication:** dedupe on `(name, address, category)` — if a record appears in both datasets with the same name + address + category, keep the one with more complete hours data.

**Refresh cadence (v1):** manual re-import when data feels stale (monthly recommended). Automated refresh is a v2 concern.

**Note:** `hours_json` is stored as-is from the source for future use but is NOT evaluated to compute open/closed status in v1. Status is a one-time enum set at import.

**Hours field quality fallback:** NYC Open Data hours fields are historically sparse. If fewer than 60% of records have usable hours data after the audit in Step 1, the import script sets `availability_status = 'unknown'` for all records without hours rather than guessing. Unknown-status pins display a gray question-mark state on the map. The "Open per listed hours" toggle excludes unknown-status locations. This is acceptable for v1 — an honest "unknown" is less harmful than a wrong "open." If coverage is very low (< 30%), reconsider whether the availability toggle should ship at all in v1 and surface this as a decision before building it.

---

### Supabase Auth and RLS

- `offerings` table: public read via anon key (no auth required to search). RLS policy: `SELECT` allowed for `anon` role; `INSERT`/`UPDATE`/`DELETE` restricted to `service_role` only (import script uses service role key).
- `pulse` table: public insert via anon key (no auth required to submit). RLS policy: `INSERT` allowed for `anon` role; `SELECT` allowed for `anon` role on `status = 'visible'` rows only. No anon `UPDATE`/`DELETE`. Flagging is done via a Postgres RPC function (not a direct UPDATE) to enforce atomic increment logic.
- Supabase anon key is safe to expose in the frontend for these policies. Service role key is server-only (import script, admin operations).

### Tech Stack

- **Frontend:** Next.js (App Router)
- **Map:** **Mapbox GL JS** — chosen over Leaflet for pin clustering, color layer rendering, and the future gap heatmap. Note: Mapbox free tier covers 50,000 map loads/month; costs become material (~$500+/month) above ~100K MAU. If budget is a concern at scale, evaluate MapLibre GL + Protomaps self-hosted tiles as a fallback path.
- **Database:** Supabase (Postgres + PostGIS)
- **Rate limiting:** Vercel middleware + Upstash Redis
- **Hosting:** Vercel

## Open Questions

1. **Availability freshness (v2):** What is the v2 mechanism for keeping hours accurate — org self-service portal, community flagging system, or periodic re-scrape? Must be decided before v2 starts.
2. **Language support:** Is English-only acceptable for v1, or is this a launch blocker for the actual target user?
3. **Gap closure loop (v2):** Can an org see a cluster of "need childcare" posts and respond with a new offering? That feedback loop is the product's long-term value — design TBD for v2.

## Success Criteria (v1 engineering gates)

- p95 time-to-first-pin rendered < 3 seconds on a mid-range mobile device on 4G (testable via Lighthouse or manual timing from cold load)
- A resident can submit an anonymous Pulse post in under 60 seconds with zero account creation
- Every displayed availability status shows a "data as of [date]" freshness indicator
- At least 100 real NYC service locations render on the map at launch

**Post-launch monitoring targets (not engineering gates):**
- At least 1 Pulse signal received per borough within 2 weeks of launch
- Zero moderation backlog older than 24 hours
